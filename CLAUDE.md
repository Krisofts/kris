# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

KRIS is a Rust CLI coding assistant, modeled after Claude Code, built with phones under Termux in mind as much as any other machine. It is **online-only**: every request goes to Gemini, Claude, OpenRouter, Opper, or OpenCode Zen (no local-inference mode exists). It has its own tool-calling agent loop, a REPL, and a set of file/git/shell tools — it is not a wrapper around another agent SDK.

## Commands

```
cargo build --release       # binary at target/release/kris
cargo test                   # unit tests (colocated in every module) + tests/agent_integration.rs
cargo test --lib <name>      # run a single unit test by name (substring match)
cargo test --test agent_integration <name>   # run a single integration test by name
cargo clippy --all-targets   # must be clean before shipping any change
cargo fmt                    # must be run before shipping any change; cargo fmt --check to verify only
```

All of the above run without needing a real model or API key — the streaming/tool-call parser, diff renderer, and filesystem tools are exercised with unit tests and a hand-rolled mock HTTP/SSE server (`tests/agent_integration.rs`) standing in for Gemini/Claude/OpenRouter/etc. Exercising a real end-to-end conversation (`kris "..."` actually talking to a model) needs a real API key for whichever provider is active.

There is no separate lint config beyond clippy defaults, and no separate formatting config beyond `rustfmt` defaults.

## Architecture

### Request flow

`main.rs` loads `Settings` (config.rs) and hands off to `repl.rs`, either `run_interactive` (REPL loop) or `run_once` (single prompt, one-shot). Everything downstream of that is per-turn:

1. `repl.rs::ask_with_iterations` → `run_turn` builds an `Agent` (`server::client_for(&settings)` picks the `ModelClient` for whichever `Provider` is active) and calls `Agent::run`.
2. `agent.rs::Agent::run` is the actual agentic loop: pushes the user message, calls `ModelClient::chat_stream`, and for each returned tool call, executes it via `ToolRegistry` and appends the result back into history, looping until a plain-text final answer or `max_iterations` is hit. Every iteration's tool-call signature (name+args) is recorded, and a stuck round is caught two ways: an exact repeat of the immediately-prior call (`detect_repeating_cycle`'s period-1 case, handled inline), or a longer oscillating cycle - the model alternating between the same 2-4 distinct calls for 3+ full repetitions without ever converging on an answer, which a single-previous-call comparison alone could never catch since each individual comparison only ever sees a *different* immediately-prior call. Neither one stops the turn cold on the first sign of trouble: the outer `'turn` loop auto-continues up to `MAX_AUTO_CONTINUES` (2) times - pushing a `Message::user` nudge (try a different approach / wrap up if there's enough info already) and giving the model a fresh `max_iterations` budget - before finally giving up with a notice. `call_signatures` is never reset across these rounds, so a relapse into the same cycle right after being nudged is still caught immediately rather than buying another full 3 repetitions. The one case that's never auto-continued is the context-budget-exceeded notice (`enforce_context_budget` returning `true` for the current turn) - more steps in the same oversized turn can't help, only a smaller next request from the user can.
3. `client.rs::ModelClient` is where the two very different wire protocols get unified into one `StreamOutcome` (content + tool_calls + prompt_tokens): `chat_stream_openai` (every OpenAI-compatible provider — Gemini/OpenRouter/Opper/OpenCode Zen all share this path, differing only in base URL/model/key) and `chat_stream_anthropic` (Claude's native Messages API — separate content-block-based streaming format, no leaked-tool-call heuristics needed since Claude never leaks a tool call into plain text).

### Provider abstraction is two-layered

`config::Provider` has five variants (`Gemini`, `Claude`, `OpenRouter`, `Opper`, `Opencode`), each with its own URL/model/API-key/context-size settings fields — but they only ever map down to two `client::Backend` variants (`OpenAiCompat`, `Anthropic`). Adding a new OpenAI-compatible gateway provider means adding a `Provider` variant plus config fields and wiring it through `server::client_for`/`describe_mode`/etc. in `repl.rs` — it does **not** need any new code in `client.rs`, since `Backend::OpenAiCompat` already covers it. `Provider::default()` is `OpenRouter` (needs the least commitment to try: one key, many models, including `:free`-suffixed ones).

### Context budget enforcement

`agent::enforce_context_budget` runs before every iteration inside the loop, not just once per turn. It extrapolates the current prompt size from the last exact `prompt_tokens` the provider reported (no network round trip - before the first such report it falls back to a plain chars/4 heuristic), and drops the *oldest complete turns* from history when over budget — but it can never touch the current in-progress turn. It takes `turn_start` (the index of the current turn's own initiating user message, tracked by `run()` since before auto-continue existed) as the actual turn boundary rather than re-deriving it from "the last `Role::User` message" - auto-continue's own nudges are also `Role::User` but belong to the *same* turn, not a new one, so naively using the last user message would let a nudge get mistaken for a fresh turn boundary and trim the real original request out from under itself. `turn_start` is adjusted in place by exactly how many messages a drop removes, so it stays valid across repeated trims within one turn. If the current turn's own accumulated tool output alone exceeds the budget (nothing older left to trim), it returns `true` and `run()` stops the turn with a clear message instead of sending a doomed, oversized request to the provider.

### Streaming robustness (client.rs)

There is deliberately no blanket `.timeout()` on the `reqwest::Client` — that would bound the entire request including body streaming, which is wrong for a slow-but-still-progressing reasoning model. Instead: `connect_timeout` bounds only the TCP handshake, and a per-chunk inactivity timeout (`stream_inactivity_timeout`, overridable via `with_stream_inactivity_timeout` for tests) inside the SSE read loop catches a stream that's gone genuinely silent. A held-back/live state machine (`apply_content_delta`) buffers content that *looks* like it might be a leaked tool call (starts with `{`, `` ` ``, or `<`) until either it resolves into a real tool call or a byte cap (`MAX_HELD_BACK_BYTES`) is hit — this only matters on the OpenAI-compatible path, since a model without reliable grammar-constrained tool-calling (confirmed on-device via a gateway-routed model) can leak a tool call as plain text; Claude's path never needs it.

### Tools

`tools/mod.rs` defines the `Tool` trait and `ToolRegistry`, which exposes tool schemas in three different shapes: `describe_all` (full JSON Schema, the base every OpenAI-compatible provider expects), `describe_all_gemini` (sanitized subset, wrapped OpenAI-style, for stricter validators), `describe_all_anthropic` (sanitized subset, Claude's flat shape). Each tool lives in its own file under `tools/`: `fs.rs` (read/list/tree/find/search — all git-ignore-aware via the `ignore` crate), `edit.rs` (write/append/edit/delete/move/mkdir — all share one confirmation gate via `Rc<Cell<bool>>` so approving "always" once covers every mutating file tool for the session), `run_command.rs` (shell exec with a 120s timeout and non-blocking output capture for backgrounded processes), `git.rs` (read-only inspection + a separate confirmed commit tool), `outline.rs` (regex-based top-level-definition scan), `ask.rs` (lets the model pause and ask the user to pick between options via the same `picker.rs` widget the `project` command uses).

When walking the filesystem with `ignore::WalkBuilder`, do not pass `min_depth()` to exclude the walked root from results — the `ignore` crate only loads a directory's own `.gitignore` when its `WalkEvent::Dir` is processed, and `min_depth` suppresses that event for the root, silently disabling the root's own `.gitignore` rules for its direct children (this bit `tree()` in production: it exploded to hundreds of thousands of tokens on an ordinary git project because `/target` in the root `.gitignore` was never actually applied). Filter out the depth-0 entry from the collected results afterward instead.

### Live turn-progress display (spinner, tool-call header, running tally)

`repl.rs::spin()` runs on its own tokio task for the whole turn and draws a "still working" line for as long as `waiting` is true - it goes quiet instead of drawing over a y/N confirmation prompt (`AWAITING_CONFIRMATION`) or `run_command`'s own synchronous status line (`COMMAND_RUNNING`), since both would otherwise race over the same terminal line. Before a reasoning model's hidden trace shows up on the wire, it shows a plain elapsed counter (`format_wait_elapsed`: "Ns" under a minute, "Xm Ys" past it); the moment `client.rs::chat_stream_openai` actually observes a `reasoning`/`reasoning_content` delta (never its content, just that it exists) it fires `on_reasoning` once, threaded through `Agent::run` the same way `on_activity` already is - the spinner then switches to a livelier cyan "Reasoning..." (animated ellipsis, no elapsed number) instead of repeating the same reassurance once there's already real proof of life. `reasoning_started` (an `Arc<AtomicBool>`) resets to false every time `waiting` is re-armed for the next iteration, so it never leaks across iterations.

`tools::{ToolCategory, tool_category, TurnCounts, SharedTurnCounts}` (moved here from repl.rs so `run_command.rs` can read the same live tally, not just repl.rs) back the "Read N files · Edited N files · Ran N commands" recap. `ToolRegistry` owns one `Arc<SharedTurnCounts>` for its whole lifetime (exposed via `ToolRegistry::turn_counts`/`Agent::turn_counts`) and hands a clone to `RunCommandTool` at construction time - `repl.rs::run_turn` fetches that *same* instance via `agent.turn_counts()` instead of constructing a second, disconnected one, so `run_command`'s own polling loop (`command_status_label`) can show this turn's real tally live while a slow command runs, instead of the screen going blank for however long that takes (which is what `COMMAND_RUNNING` otherwise leaves it doing, since it silences repl.rs's own spinner for that whole stretch). The tool-call header's leading "●" (`tool_bullet`) is colored by the same `ToolCategory` - blue for a read, green for an edit, yellow for a command - instead of one undifferentiated cyan, and a write/edit/append result's trailing "(+N -M)" diff stat (`colorize_diff_stat`) is recolored green/red within its otherwise-dim summary line, matching `diff.rs::render_unified_diff`'s own per-line coloring.

### Session persistence

`session_store.rs` saves `Session::history` to `~/.config/kris/sessions/<sanitized-root-path>-<hash>.json` after every turn (`repl.rs::run_turn`, right after `agent.run` returns, regardless of Ok/Err - a failed turn can still have kept real progress) and loads it back in `Session::new`/`refresh_root`. Keyed by project root, not a single global file, so switching projects with `project <name>` resumes *that* project's own last conversation rather than sharing one history across all of them. Each session file also carries its own `root` and a `starred` flag (a `PersistedSession` envelope, not just the raw history array) so `session_store::list_sessions()` can enumerate every saved project without having to reverse the sanitized filename back into a path - that's what backs the `resume` REPL command (KRIS's counterpart to Claude Code's own `/resume`: picks a saved session via `picker.rs` and switches straight to it via `Session::switch_to_root`, which repoints `workspace` itself since a saved session's project isn't necessarily under the *current* workspace), as well as `star` and `delete-session`. `star [name]` toggles `starred` via `session_store::set_starred` (`save()` always reads it back off disk first and carries it forward, since `save()` runs after every turn and would otherwise silently reset it) - starred sessions sort first in `list_sessions()` and show a leading "★" (`repl.rs::session_label`, shared by all three commands' pickers). `delete-session [name]` permanently removes a saved session from disk (`session_store::clear`, the same function `clear` already used for wiping just the *current* session) after a y/N confirmation - unlike `clear`, it can target any saved session, not only the active one, and clears it from the in-memory `Session` too if it happened to be the one just deleted. All three commands share `pick_session`/`find_session_by_name`: an empty argument shows `picker.rs`'s arrow-key picker (falling back to a plain numbered list on a non-interactive terminal), a non-empty one matches directly by the project folder's own name, the same shortcut `project <name>` already offers. `export` (KRIS's counterpart to Claude Code's `/export`, since Claude Code has no `/handoff`) renders history as readable Markdown via `export.rs` - a human-facing format, deliberately separate from `session_store`'s JSON, which only KRIS itself ever needs to read back. A missing or corrupt session file is never an error - `load` just falls back to an empty history. Tests that point `$HOME` at a scratch tempdir (session_store.rs and repl.rs both have one) must lock `crate::test_support::HOME_ENV_LOCK` - it's shared crate-wide on purpose, since `$HOME` is a single process-global value and a per-module lock alone doesn't stop one module's test from repointing it out from under another module's test running concurrently on a different thread.

`compact` (KRIS's counterpart to Claude Code's own `/compact`) is a different way of getting back under the context budget than `enforce_context_budget`'s automatic trimming: instead of silently dropping the oldest turns outright, it calls `Agent::summarize` - a plain-text-only request (`tools: None`, so the model can never call one) asking for a recap - then replaces `history` with just `[system_prompt, recap]` (`repl.rs::build_compacted_history`). The system prompt has to be reconstructed explicitly here (`Agent::system_prompt` is `pub(crate)` for exactly this) so the compacted history is non-empty and `Agent::run`'s own `history.is_empty()` check doesn't insert a second one on the next turn. Leaves the original conversation untouched if the summarization call itself fails, so a network hiccup can't destroy history in exchange for nothing.

### Project conventions (KRIS.md)

`kris_md.rs::seed_new_project` writes a project's own `KRIS.md` from a template the moment it's resolved as the active root (`Session::new`/`refresh_root`) - but only if the directory is completely empty and has no `KRIS.md` yet, so an existing project is never touched uninvited (it might already have its own project-specific one from `init`). The template itself is `~/.config/kris/KRIS.md` if the user has customized one, otherwise a built-in default (house rules: keep files under 300 lines, always run + smoke-test before finishing). `Agent::run` folds a project's `KRIS.md` (if it has one) straight into the system prompt on the first turn of a session (not a separate message, and not re-read every turn) - a project's conventions actually steer the model every session this way, instead of sitting there as a file it only sees if it happens to `read_file` it on its own initiative.

### Message representation

`message.rs`'s `Message`/`ToolCall` are OpenAI-shaped internally (used for every OpenAI-compatible provider directly). `client.rs::to_anthropic_messages` converts that into Claude's native shape (system prompt pulled into its own field, content as typed blocks, consecutive same-role messages merged since Claude rejects repeats) only at the point of sending a Claude request — the rest of the codebase (agent loop, context budget, REPL) only ever deals with the OpenAI shape.

### Git workflow for this repo specifically

Development happens on a working branch; `main` is only fast-forward-merged into when explicitly asked. Before merging: `git fetch origin main <branch>`, confirm a clean fast-forward is possible (`git rev-list --left-right --count origin/main...origin/<branch>` showing 0 behind), merge with `--ff-only`, push, then switch back to the working branch.

### Verifying a fix before shipping it

The established discipline in this codebase: when fixing a bug, reproduce it first (a minimal script or a temporarily-reverted patch that makes the new regression test fail with the exact same symptom), then apply the real fix and confirm the same test now passes. Every bug fix ships with a regression test that encodes this proof, not just a patch.
